1、瀑布模型的每个阶段输出物，缺点？
 + 计划---产品的计划，SRS
        + 需求分析---评审报告，SRS
        + 设计----HLD LLD
        + 编码----产品
        + 测试----测试计划，测试方案，测试用例，测试报告
        + 运行维护----用户手册（实施），运维手册（开发）
        + 缺点
            + 测试介入较晚，如果发现是需求/计划的问题，修复成本非常高
            + 之前的软件很小，功能不复杂，开发简单，现在的软件很复杂
            + 由于是线性的用户只能在末期才能看到结果，增加风险
            + 不利于需求的变更
            + 是只有将产品做完成后才一次性交付给用户
2、敏捷模型的特点？
 + 快，版本迭代快
        + 将一个大项目分成很多可以独立运行的小项目，单独开发上线
        + 及时响应市场变化---可以变更
        + 迭代周期较短一般为2周左右
3、V和W模型相比有何优缺点
	+ 线性的，测试在后期代码出来才进行的测试工作，如果前期有问题修复成本很高--同瀑布
    + 测试活动严重滞后于研发活动
    + 适用于周期较短的项目
    + 单线流程，如果研发故障，流程直接终端
    + v模型主要测试对象是代码产品

5、缺陷的流程管理
	+ 在测试过程中，发现一个Bug，测试人员在缺陷管理工具中新建缺陷报告，指派给对应的开发人员，开发人员确认后
	[如果是一个Bug]进行修复，再指派给对应的测试人员，测试人员进行回归测试，如果Bug确定已经修复完成则直接关闭，如果还没有修复完成/引入了新的Bug
	继续驳回，甚至提交新的Bug

7、软件测试流程具体是什么 怎么进行的
+ 首先我们拿到需求文档先熟悉需求，了解具体项目的一个流程，然后参与需求评审，明确具体的功能流程和一些细节的问题，再
	+ 使用Xmind工具对自己负责的模块进行拆分提取对应的测试点，编写测试方案，再根据Xmind编写出测试用例，在组内进行用例
	+ 评审工作，确保用例的覆盖度和有效性，待开发提测后，首先进行冒烟测试，确保主业务流程通过，在冒烟测试通过后，进行
	+ 系统的全面测试，伴随缺陷的跟踪回归，达到一个测试的通过标准后，输出测试报告，联系用户进行UAT测试，待通过后进行上线。

 + 上线----产品发布

        + 用户量少------一般是0-4点
        + 休息好
        + 回滚
        + 提前准备好测试资源，以及数据备份

                + 需求覆盖度达到100%

11、测试计划和测试方案的区别
　所谓测试计划是指描述了要进行的测试活动的范围、方法、资源和进度的文档。它主要包括测试项、被测特性、测试任务、谁执行任务和风险控制等。
- 　　　所谓测试方案是指描述需要测试的特性、测试的方法、测试环境的规划、测试工具的设计和选择、测试用例的设计方法、测试代码的设计方案。
12、测试用例的内容、优先级定义，以及如何编写
具体的内容
        + 用例编号/用例id
            + A-B-C-D格式：  qq-ST-login-001
        + 测试项/所属模块
            + 一般按照菜单来划分，有大模块和子模块之分
        + 用例标题
            + 明确测试的内容，简单明了
        + 用例属性
            + 一般为功能测试，界面测试，兼容性等
        + 重要级别/优先级
            + 一般分为4级
            + 1/极高-----用于冒烟，主要是主业务流程，一般一个模块就一条
            + 2/高  -----一般用于主业务流程非冒烟
            + 3/中  ----一般业务流程和界面的常规输入情况
            + 4/低  ---- 主要为一些异常的输入输出等情况
        + 预置条件
            + 一般在此存放需要的资源，登陆的账号密码，需要提前进入那个界面，哪些图片等
        + 操作步骤
            + 模拟用户的实际操作
            + 分步骤写清楚
            + 在步骤里没有结果
        + 预期结果
            + 在没有bug的情况下出现的结果
            + 以此进行判断是否有bug
        + 适用阶段
            + 一般为冒烟和系统测试
	
- 用例编写的原则
    + 一条用例只测试一个功能点
    + 每个模块先写一条冒烟的
    + 用例中除了考虑正常的情况更重要的还需要考虑异常的情况
    + 标题不能重复，简单明了，禁止出现模糊的词
    + 在标题和操作步骤，要清晰，别人能一眼看懂测的是什么，以及具体该怎么执行
    + 涉及到输入框的首先考虑空值情况
    + 如果涉及到条件有多个，在测试其中一个的时候默认另一个满足
    + 一般界面的用例，优先级为中或者低；而且整个模块的高占很少部分

14、软件测试级别划分，以及各个之间的区别，αβ的区别
 + 需求测试----需求评审
        + 完整性
        + 正确性
        + 一致性
        + 可行性
        + 无二义性
        + 健壮性
        + 必要性
        + 可测试性
        + 可修改性----
    + 单元测试
        + 组件，函数，类，数据逻辑，模型等各种代码逻辑
        + 对应的是详细设计文档--lld
        + 可以发现内存泄漏，算法冗余，分支覆盖率低，等各种效率低的问题
        + 需要测试人员有代码基础
        + 属于白盒测试
    + 集成测试---接口测试
        + 模块之间的数据交互
            + 内部接口和第三方接口
        + 对应文档是概要设计说明书-hld
        + 可以发现一些功能上无法发现的bug---转账转负数
        + 属于灰盒测试
    + 系统测试
        + 就是对整个系统的全面测试，包括很多项，大类包括，功能，兼容，性能，安全，用户体验等等方面的测试
        + 对应需求文档--SRS
        + 快，方便操作，主观上已于用户接受
        + 一般在市场上，通常将系统测试理解为功能测试？？
        + 属于黑盒测试
        + 注意：
            + 配置项测试：
                + 有软件提供方组织，由独立于软件开发人员实施
                + 系统测试由软件需求方组织，由独立于软件开发人员实施
                + 二组都可以委托第三方进行测评
    + 验收测试
        + α
            + 开发环境，
            + 内测
            + 通常开发在场
        + β
            + 用户环境
            + 公测
            + 通常开发不在场

- 测试方法
    + 是否关注内部逻辑
        + 白盒
            + 关注内部逻辑
            + 需要有一定的代码基础
            + 能发现深层次的问题
            + 对象：设计文档，模块，代码
        + 灰盒
            + 白+黑
            + 接口的测试
            + 进行集成测试关注日志
        + 黑盒
            + 只关注输入输出不关注内部逻辑
            + 快，执行很快
            + 主要在项目开始的时候，整个项目的测试皆可以
            + 用户易于接受
            + 不能发现深层次的问题
            + 对象包括：系统，SRS，手册
    + 是否执行代码
        + 静态----看
            + 设计文档，SRS，手册，代码
        + 动态
            + 模块，系统
    + 专项
        + 静态黑盒
            + SRS，手册
        + 静态白盒
            + 设计文档，代码
        + 动态黑盒
            + 系统
        + 动态白盒
            + 模块
    + 是否为人执行
        + 手工
            + 最重要的还是手工，提测后肯定会用手工
            + 快，能进行发现问题
            + 灵活，手工可以发现更多的问题
        + 自动化
            + 为了方便执行，编写自动化脚本，运行脚本进行验证此功能，测的内容都在脚本中会写好
            + 分类
                + 接口自动化-----接口已开发结束就会进行
                + ui自动化  -----系统以及稳定，每次都需要进行验证某个重要的功能，但是此功能又没有发生改变
            + 遵循杀虫剂悖论
            + 无法发现新的bug

请按照用例标准格式编写
   购物车需求
   有以下功能
   1.添加商品
     购物车中无商品时，可以添加一个或多个商品到购物车，可以多次添加同类商品，也可以添加一类商品的多个数量。
