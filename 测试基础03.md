## 测试模型
- V模型
    + 线性的，测试在后期代码出来才进行的测试工作，如果前期有问题修复成本很高--同瀑布
    + 测试活动严重滞后于研发活动
    + 适用于周期较短的项目
    + 单线流程，如果研发故障，流程直接终端
    + v模型主要测试对象是代码产品
- W模型（双V模型）
    + 相对于v模型，多了对各种文档的测试，包括需求文档，设计文档等；
    + 可以在早期发现问题，解决v的问题1
    + 串行和并行的问题
    + 滞后于研发
- X模型
    + 测试活动的滞后问题
    + 探索性测试--根据经验
- H模型
    + 独立于开发0
	+ 只要达到测试就绪点，就能开展测试
- 敏捷测试模型
- 前置测试模型
    + 验收测试和技术测试保持相互独立
    + 将测试和开发紧密结合的模型
## 测试基础
- 软件测试以及做好一个测试需要哪些技能
    + 检测软件是否合格
        + 合格的标准
            + 符合用户的需求
    + 专业技术，工具的使用，测试的经验，沟通。。。。
    + 测试目的
        + 发现被测试对象与用户需求之间的差异
        + 发现并解决缺陷，增加人们对软件质量的信心
        + 提供数据依据
            + 测试是否通过
                + 需求覆盖度达到100%
                + 12级用例执行100%，3级执行95以上，4级执行80%以上
                + 12级bug修复100%，3级bug修复95%以上，4级bug修复80%
                + 具体在用户验收的时候，还要和用户进行沟通确认
        + 预防缺陷，降低风险
        + 尽可能多的找出bug
        + 保证产品正常上线
        + 保证产品质量，维护和客户的长期合作。。。。。
    + 测试阶段
        + 需求分析----测试需求文档
        + 软件设计----测试设计文档
        + 编码开发----测试产品
        + 验收-----测试产品
        + 运营维护----测试产品和一些文档

- 缺陷报告
    + 缺陷id----在实际中，会自动生成的
    + 概要描述（缺陷标题）---- 要有意义，一目了然
    + 发现人，一般指测试人员，用户
    + 发现时间
    + 修复截止时间
    + 下级处理人/指派人员/修复人员----一般指开发人员
    + 所属版本---为了方便定位
    + 所属模块---为了方便定位
    + 缺陷状态---使用的alm/qc工具中的状态
        + new，测试新建
        + open，测试打开
        + fix，开发修复
        + close，测试关闭
        + reopen，重新打开
        + reject，开发驳回
        + 缺陷管理流程是怎么样的？？？
    + 缺陷严重程度
        + low，medium，high，very high，urgent
        + 1-5，1-4
        + 高中低
        + a-e
        + 致命，严重，一般，提示
            + 致命：系统崩溃，软件崩溃闪退，核心业务流程无法正常运行，内存泄漏等
            + 严重：重要功能，子功能流程缺陷
            + 一般：常规的显示错误，界面类的一般错误
            + 提示：体验，布局问题，一般是一些建议问题，提示语不友好等
    + 修复优先级---和严重程度成正比
    + 详细描述----细
        + 具体复现的步骤结果和预期结果
        + 按步骤写清楚
    + 附件
        + 报错截图
        + 日志
        + 数据表的截图
        + 数据流量包

## 作业
- 缺陷报告的类容，你认为哪些比较重要
- 严重程度如何划分
- 简述bug周期（缺陷管理流程）
	+ 在测试过程中，发现一个Bug，测试人员在缺陷管理工具中新建缺陷报告，指派给对应的开发人员，开发人员确认后
	[如果是一个Bug]进行修复，再指派给对应的测试人员，测试人员进行回归测试，如果Bug确定已经修复完成则直接关闭，如果还没有修复完成/引入了新的Bug
	继续驳回，甚至提交新的Bug
- 如果开发不认可你提的bug，你怎么处理
	+ 首先和开发沟通确认不认可的原因
	+ 如果是需求理解不一致----产品沟通
	+ 需求没有具体的明确说明----属于优化类的问题
	+ 在测试环境有问题，在开发环境没有复现，----以测试环境为准（测试环境基本和用户一致)
	+ 开发人员太忙----催
	+ 不想改----大保健
