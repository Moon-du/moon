- 测试类型
    + 功能
    + 性能
        + 压力
            + 饱和状态下的持续运行时间，系统状态，持续时间
            + 指标有错误
            + 会耗尽系统资源
        + 负载
            + 在临界条件下，找临界点，没有时间概念(并发测试)
            + 所有指标都正常的
        + 容量
        + 可靠性--稳定性
            + MTBF（app的稳定性）  MTTR
    + 安全
        + 使用工具，appscan
    + 兼容性---主要是软件
        + c/s
            + 操作系统，配套软件，分辨率。。。
        + b/s
            + 不同浏览器，分辨率，操作系统......
    + 可用性测试---用户体验性
    + 移植性----主要针对硬件
    + 维护测试---改正性，完善性，移植性，预防性
    + 确认测试---验证开发改后的bug是否修复成功
    + 回归测试---是否引入新的bug（基于风险，基于操作，基于缺陷）
        + 选择性回归/部分回归
        + 全部回归/完全回归
- 测试方法
    + 是否关注内部逻辑
        + 白盒
            + 关注内部逻辑
            + 需要有一定的代码基础
            + 能发现深层次的问题
            + 对象：设计文档，模块，代码
        + 灰盒
            + 白+黑
            + 接口的测试
            + 进行集成测试关注日志
        + 黑盒
            + 只关注输入输出不关注内部逻辑
            + 快，执行很快
            + 主要在项目开始的时候，整个项目的测试皆可以
            + 用户易于接受
            + 不能发现深层次的问题
            + 对象包括：系统，SRS，手册
    + 是否执行代码
        + 静态----看
            + 设计文档，SRS，手册，代码
        + 动态
            + 模块，系统
    + 专项
        + 静态黑盒
            + SRS，手册
        + 静态白盒
            + 设计文档，代码
        + 动态黑盒
            + 系统
        + 动态白盒
            + 模块
    + 是否为人执行
        + 手工
            + 最重要的还是手工，提测后肯定会用手工
            + 快，能进行发现问题
            + 灵活，手工可以发现更多的问题
        + 自动化
            + 为了方便执行，编写自动化脚本，运行脚本进行验证此功能，测的内容都在脚本中会写好
            + 分类
                + 接口自动化-----接口已开发结束就会进行
                + ui自动化  -----系统以及稳定，每次都需要进行验证某个重要的功能，但是此功能又没有发生改变
            + 遵循杀虫剂悖论
            + 无法发现新的bug
- 禅道
    + 测试主要在此系统  编写用例，执行用例，提bug，看数据分析结果
    + 缺陷状态：激活，已解决，已关闭
## 测试流程
- 阶段划分
    + 测试计划
        + 明确测试的范围，测试的时间，测试通过的标准，人力，资源，风险等
            + 测试是否通过
                + 需求覆盖度达到100%
                + 12级用例执行100%，3级执行95以上，4级执行80%以上
                + 12级bug修复100%，3级bug修复95%以上，4级bug修复80%
                + 具体在用户验收的时候，还要和用户进行沟通确认
        + 输出---测试计划文档
        + 一般由测试经理编写---测试负责人
    + 测试设计
        + 确定测试使用的方法，类型，具体使用的测试工具，缺陷流程的确定，优先级和严重程度的定义，用例的设计方法.....
        + 输出测试方案（测试策略）
        + 测试人员编写（测试组长）
    + 测试实现
        + 搭建测试环境，用例的编写和评审，测试数据的构造
        + 输出测试用例（测试案例），测试环境
        + 测试人员
    + 测试执行
        + 执行用例，缺陷跟踪回归，测试报告的确定
        + 输出---测试报告
        + 测试人员/测试组长
