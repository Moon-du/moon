- 测试方案
    + 冒烟测试：测试人员专门针对开发刚提测的软件，对其重要的功能流程的测试，一般选择优先级高的用例进行执行，目的是再级短的时间内确定功能是否达到转测标准，一般只有在冒烟测试通过后才进行全面测试。
    + 包括，测试的方法，类型，测试对象，用例方法，用例id的确定，缺陷流程严重程度确定。。。
- 测试报告
    + 测试的结论，本次测试的资源汇总统计，用例统计分析，缺陷统计分析，遗留问题。。。。

## 测试设计
- 质量分类
    + 过程质量-----生产流程是否规范
    + 内部质量-----静态测试是否合格
    + 外部质量-----功能。性能表现
    + 使用质量-----用户体验性
- 软件质量特性
    + 功能性------适合性，准确，互操作，保密安全，功能依从
    + 可靠性------成熟，容错，易恢复性，可靠依从
    + 易用性------理解，易学，易操作，吸引，依从
    + 效率  ------时间，资源，依从
    + 可维护------分析，改变，稳定。。。
    + 可移植------。。。。
- 测试用例设计
    + 测试用例/测试案例/test case---测试使用的一个例子
        + 对某一特定的目标，针对其预期的功能进行验证，将验证的步骤和各种操作写入一个具体的excel中方便测试人员进行执行，通过执行结果和预期结果进行对比，如果一样表示本功能测试通过，如果不一样表示此功能有bug
    + 具体的内容
        + 用例编号/用例id
            + A-B-C-D格式：  qq-ST-login-001
        + 测试项/所属模块
            + 一般按照菜单来划分，有大模块和子模块之分
        + 用例标题
            + 明确测试的内容，简单明了
        + 用例属性
            + 一般为功能测试，界面测试，兼容性等
        + 重要级别/优先级
            + 一般分为4级
            + 1/极高-----用于冒烟，主要是主业务流程，一般一个模块就一条
            + 2/高  -----一般用于主业务流程非冒烟
            + 3/中  ----一般业务流程和界面的常规输入情况
            + 4/低  ---- 主要为一些异常的输入输出等情况
        + 预置条件
            + 一般在此存放需要的资源，登陆的账号密码，需要提前进入那个界面，哪些图片等
        + 操作步骤
            + 模拟用户的实际操作
            + 分步骤写清楚
            + 在步骤里没有结果
        + 预期结果
            + 在没有bug的情况下出现的结果
            + 以此进行判断是否有bug
        + 适用阶段
            + 一般为冒烟和系统测试
    + 设计方法
        + 黑盒
            + 等价类---
                + 有效等价----写用例时从范围类选择一个就行
                + 无效等价----选择一个
                + 代表，输入框要求只能输入数字，字母等类型的，范围类的
            + 边界值
                + 考虑3个点，上点，离点，内点
                + 特别是范围类的如6-18 等有上限或者下限的首选
            + 判定表
            + 因果图
            + 正交实验
            + 状态迁移
            + 场景设计 
        + 白盒

- 用例编写的原则
    + 一条用例只测试一个功能点
    + 每个模块先写一条冒烟的
    + 用例中除了考虑正常的情况更重要的还需要考虑异常的情况
    + 标题不能重复，简单明了，禁止出现模糊的词
    + 在标题和操作步骤，要清晰，别人能一眼看懂测的是什么，以及具体该怎么执行
    + 涉及到输入框的首先考虑空值情况
    + 如果涉及到条件有多个，在测试其中一个的时候默认另一个满足
    + 一般界面的用例，优先级为中或者低；而且整个模块的高占很少部分
